Patterns
Design Rules
Non-negotiable constraints. These govern every screen, every component, every decision.
✓
One primary action per screen
Each page has one orange button. The FAB counts. The "Save" button in a form counts. Never two orange buttons competing.
✓
Barlow Condensed Bold + uppercase for ALL labels
Table headers, form labels, tile labels, badge text, button text, filter chips — all condensed, bold, uppercase. This is the system voice.
✓
Form panels slide from the right
All create/edit flows use translateX(100%) → 0. No modals. The list stays in place beneath. Back button always present.
✓
Tables over cards for tabular data
If there are 3+ columns of data, it's a table. Cards are for single records (CAPA detail, legal register entry). Never both on the same screen.
✓
Min 44px touch targets everywhere
Buttons, tab items, filter chips, FAB. Operators use these with gloves. No small tap targets.
✗
No modal dialogs for forms
Modals clip content on mobile and fight the browser back button. Always use slide-in panels or inline expansion.
✗
No decorative orange — orange = action
Orange left borders on tiles are structural (navigation affordance). Orange used as decoration elsewhere dilutes the primary signal.
✗
No gradient text or glassmorphism
Both read as consumer-app softness. This is a QMS tool. Clarity over style.
✗
Never animate layout properties
No transitions on width, height, top, left, or margin. Use transform and opacity only. Performance on low-end Android devices matters.
✗
No raw pixel values in CSS
Every spacing value must reference --sp1 through --sp10. Every color must reference a token. No inline style="padding:16px" in production code.
